# 記事レビュースキル

このスキルは、テックブログ記事のレビュー方法を定義します。
記事のレビューを依頼された際に参照してください。

**対象記事の種類:**
- プログラミング技術記事（チュートリアル、技術解説、トラブルシューティング、ベストプラクティス）
- 論文解説記事（機械学習、コンピュータサイエンスなど）

---

## 1. 良い記事の条件

レビューの目的は、以下の3軸を満たす記事に仕上げることです。

- **正確性**: 技術情報・論文の内容を正確に伝える
- **わかりやすさ**: 読者がストレスなく理解できる構成と表現
- **実用性**: 読者が読み終えた後に何かを得られる、活かせる内容

---

## 2. チェック基準

### 2.1. 技術的な正確さ（優先度: MUST）

**プログラミング系記事:**

- [ ] 技術・ライブラリ・フレームワークの情報は正確か
- [ ] バージョン情報が明記されているか（重要な場合）
- [ ] 事実に基づかない、誤解を招く表現はないか
- [ ] コードサンプルは文法的に正しく、実行可能か
- [ ] 外部リソース（公式ドキュメントなど）への参照は正確か
- [ ] 非推奨のAPIや古い情報を使っていないか

**論文解説記事:**

- [ ] 論文の基本情報（著者・タイトル・学会/ジャーナル・年度）が正確に記載されているか
- [ ] 論文のDOIやarXiv IDが記載されているか
- [ ] 論文の主張や結論を正確に理解・解釈しているか
- [ ] 数式の転記に誤りがないか
- [ ] 実験結果やベンチマークデータの引用が正確か
- [ ] 自分の解釈と論文の記述を明確に区別しているか

**良い例（プログラミング系）:**

```markdown
## Node.jsでのファイル読み込み

Node.js v18以降では、`fs/promises`を使った非同期処理が推奨されています。

\```javascript
import { readFile } from 'fs/promises';
const data = await readFile('./example.txt', 'utf-8');
\```

> 参考: [Node.js File system API](https://nodejs.org/docs/latest-v18.x/api/fs.html)
```

**良い例（論文解説）:**

```markdown
> Vaswani, A., et al. (2017). "Attention is All You Need".
> NeurIPS 2017. [arXiv:1706.03762](https://arxiv.org/abs/1706.03762)

論文では、Self-Attentionを以下のように定義しています（Section 3.2.1）:
$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$
```

---

### 2.2. コードの品質（優先度: MUST）

**プログラミング系記事:**

- [ ] コードは読みやすく、一貫したスタイルでフォーマットされているか
- [ ] 変数名・関数名は役割が分かりやすいか
- [ ] ベストプラクティスに沿っているか
- [ ] セキュリティ上の問題がないか（ハードコードされたパスワードなど）
- [ ] エラーハンドリングが適切か
- [ ] コードブロックに言語指定（` ```python ` など）があるか

**論文解説記事:**

- [ ] 数式はLaTeX/KaTeX形式で正確に記述されているか
- [ ] 数式の各記号の意味が説明されているか
- [ ] 実装例を示す場合、論文の定義に忠実か

---

### 2.3. 引用とクレジット（優先度: MUST）

- [ ] 外部情報源を使用する場合、適切に引用されているか
- [ ] 引用元のURLや書籍名が明記されているか
- [ ] 画像や図表の出典が示されているか
- [ ] 他者のコードを使用する場合、その旨が明記されているか

---

### 2.4. 誤字脱字と文法（優先度: MUST）

- [ ] 誤字脱字がないか
- [ ] 文法的に正しい日本語表現か
- [ ] 「ら抜き言葉」などの口語表現が混在していないか
- [ ] 敬体（です・ます調）と常体（だ・である調）が混在していないか
- [ ] 同じ語尾の連続（「〜ます。〜ます。〜ます。」）がないか

---

### 2.5. 明確性と論理性（優先度: SHOULD）

- [ ] 記事の目的や主張が冒頭で明確に述べられているか
- [ ] 文章の構成は論理的で、話の流れは自然か
- [ ] 専門用語や略語には初出時に説明があるか
- [ ] 結論や要点が明確にまとめられているか
- [ ] 前提条件・対象読者が明示されているか

**良い例:**

```markdown
## はじめに

この記事では、Go言語を使ってDDDの値オブジェクトを実装する方法を解説します。
DDDの基本概念（エンティティ、値オブジェクト）を理解している中級者向けの内容です。

### 前提環境
- Go 1.21以降
```

---

### 2.6. 読みやすさ（優先度: SHOULD）

- [ ] 全体として読者がストレスなく読み進められる構成か
- [ ] 一文が長すぎないか（目安: 60〜80文字以内）
- [ ] 冗長な表現や重複がないか
- [ ] 箇条書きや図表が効果的に使われているか
- [ ] 見出しの階層構造（h2, h3, h4）は適切か

---

### 2.7. 実用性（優先度: SHOULD）

**プログラミング系記事:**

- [ ] 読者が実際の業務や学習に活かせる内容か
- [ ] 再現可能な手順や具体的な情報が提供されているか
- [ ] 環境構築の手順や依存関係が明示されているか
- [ ] 発生しうる問題とその対処法が示されているか
- [ ] サンプルコードは完全で、そのまま動作するか

**論文解説記事:**

- [ ] 難解な概念を分かりやすく説明しているか
- [ ] 実装への応用方法や実用例が示されているか（可能な場合）
- [ ] 論文の限界や課題についても言及しているか
- [ ] 読者が元論文を読む際の理解の助けになるか

---

### 2.8. 独自性（優先度: SHOULD）

- [ ] 筆者自身の経験・考察・意見が含まれているか
- [ ] 公式ドキュメントの翻訳や情報の羅列になっていないか
- [ ] 「なぜそうするのか」の理由や背景が説明されているか

---

### 2.9. タイトルと見出し（優先度: SHOULD）

- [ ] タイトルは記事の内容を的確に表しているか（煽り表現を使っていないか）
- [ ] 見出しの階層構造（h1→h2→h3）が適切か
- [ ] 見出しだけ読んでも記事の流れが理解できるか
- [ ] 見出しが長すぎないか（目安: 30文字以内）

---

## 3. 記事種別の重点項目

記事の種類によって、重点的にチェックすべきポイントが異なります。

### 3.1. チュートリアル記事

- 手順が順序立てて説明されているか
- 各ステップでの期待される結果が示されているか
- 初心者でも理解できる丁寧な説明があるか

### 3.2. 技術解説記事

- 技術的な背景や「なぜ必要か」が説明されているか
- 他の選択肢との比較があるか
- 実用例が示されているか

### 3.3. トラブルシューティング記事

- 問題の症状が明確に記載されているか
- 原因の分析プロセスが説明されているか
- 解決策が具体的で再現可能か

### 3.4. ベストプラクティス記事

- 推奨する理由が論理的に説明されているか
- アンチパターンとの比較があるか
- 適用する際の注意点が記載されているか

### 3.5. 論文解説記事

- [ ] 論文の基本情報（著者・タイトル・学会・年度・DOI/arXiv ID）が明記されているか
- [ ] 研究の背景と動機が説明されているか
- [ ] 提案手法の核心が分かりやすく解説されているか
- [ ] 重要な数式とその意味が丁寧に説明されているか
- [ ] 実験結果の数値だけでなく、その意味・示唆が説明されているか
- [ ] 論文の新規性・貢献が明確にされているか
- [ ] 論文の限界や今後の課題にも言及されているか
- [ ] 自分の解釈と論文の記述を明確に区別しているか

---

## 4. レビュー対象外の項目

- **個人的なスタイルの選択**: 技術的に正しければ、筆者の好みを尊重
- **主観的な好み**: 「この技術が好き」といった個人の嗜好
- **記事のテーマ選定**: 筆者が書きたいテーマを自由に選ぶ権利を尊重

---

## 5. フィードバックの書き方

### 5.1. 基本方針

- **具体的に**: 「わかりにくい」ではなく、どこがどう分かりにくいかを明示
- **建設的に**: 問題点と改善案をセットで提示
- **肯定的に**: 良かった点も積極的に評価し、執筆者のモチベーションを高める（3つ以上）
- **簡潔に**: 1コメント1ポイントを原則とする

### 5.2. コメントの構造

```markdown
## レビュー結果

### 良かった点 ✅

- [具体的な良かった箇所と理由]

### 改善提案 (MUST) 🔴

- **[セクション名]**: [問題点]
  - 改善案: [具体的な修正案]

### 改善提案 (SHOULD) 🟡

- **[セクション名]**: [問題点]
  - 改善案: [具体的な修正案]

### 改善提案 (MAY) 🔵

- **[セクション名]**: [問題点]
  - 改善案: [具体的な修正案]
```

### 5.3. トーンとマナー

- 敬語を使用し、丁寧な表現を心がける
- 「〜してください」ではなく「〜すると良いでしょう」
- 筆者の努力や工夫を認める言葉を添える

---

## 6. レビュープロセス

1. **全体の把握**: まず記事全体を通読し、目的・ターゲット読者・記事種別を理解する
2. **MUST項目の確認**: 技術的正確さ・コード品質・引用・誤字脱字を優先的にチェックする
3. **SHOULD/MAY項目の確認**: セクション2.5〜2.9のチェック項目を記事種別に応じて適用する
4. **総評の作成**: 良かった点・改善提案をセクション5.2の形式でまとめる

### 完了後のチェックリスト

- [ ] MUST項目（技術的正確さ・コード品質・引用・誤字脱字）をすべて確認したか
- [ ] 良かった点を3つ以上挙げたか
- [ ] すべての指摘に改善案を添えたか
- [ ] レビューコメントの口調は丁寧で建設的か
- [ ] 優先度（MUST/SHOULD/MAY）を適切に設定したか
- [ ] 記事の種類（3.1〜3.5）に応じた重点項目を確認したか
